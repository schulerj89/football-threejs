# AGENTS.md

## Project

This repository is a low-poly 3D American football game prototype built with Three.js, Vite, TypeScript, WebGL, and a future WebGPU path. The official in-game title is `Football JS`, centralized in `src/config/GameBrand.ts`; player-facing title strings should resolve through that config rather than scattering the brand name. The long-term target is a stylized low-poly 11v11 American-football game with cinematic and broadcast-style presentation. The normal player-facing mode is now an offense-only exhibition shell: the player controls the user team's offensive possessions, while opponent possessions resolve through deterministic broadcast-style drive summaries. The score-attack mode is preserved as a development regression harness, not the final product identity. The active default prototype is the twenty-two-player 11v11 game mode with one rushing play and one passing play. The 7v7 and 5v5 modes remain explicit development and regression modes. Current presentation integration work should preserve one authoritative immutable presentation-event stream for audio, commentary, crowd, camera, captions, and holds. Team identity is handled through fictional team profiles, title-screen uniform customization, and one shared team theme consumed by player visuals, helmets, HUD, play cards, field end zones, and crowd accents. Roster identity is handled through fictional team rosters and active lineup bindings; gameplay systems continue to use stable gameplay player IDs while presentation resolves names, numbers, positions, and appearance identity through the roster layer.

The default active playable prototype is an eleven-on-eleven offense-only exhibition game with semantic data-defined formations for Inside Zone 11 and Spread Quick 11, four configurable quarters, deterministic opening and second-half possession, user offensive drives, abstract punts, simulated opponent drives, automatic extra points, halftime, final score, and game over. Receiver routes are ordered mathematical paths with gameplay-owned route progress, active segment state, completion state, sampling, tangents, projection, reset behavior, pre-snap on-field route art, development-only route auditing, and route-aware pass targeting. The explicit `?playbook=7v7` path preserves the seven-on-seven drill with Inside Zone 7, Outside Zone 7, Quick Pass 7, and Twin Slants Flat for regression coverage. The explicit `?playbook=5v5` path preserves the legacy five-on-five drill with Inside Run, Outside Run, Quick Pass, and Slant Flat for regression coverage. The prototype also includes graphical pre-snap SVG play cards generated from gameplay play data, low-poly procedural player bodies with visible low-poly heads, cloned low-poly helmet visuals, deterministic skin-tone presentation, fictional roster identities with a title/setup lineup preview, presentation-only quarterback ratings and scouting profiles, a pooled retro controlled-player label, a procedural low-poly football visual, presentation-only pregame warmup staging, a warmup-clone quarterback showcase card, a one-time interactive opening coin toss with presentation-only captains/referee and textured 3D ceremonial coin, presentation-only sideline teams and tunnel tableau staging, a presentation-only two-official crew, visual-only procedural poses and locomotion, a field generated from a pure field specification with batched static markings and presentation-only turf/yard-number/goalpost/sideline elements, a normal-game mathematical low-poly stadium bowl, selected eligible receivers on pass plays, AI blockers, AI defenders, deterministic blocking engagements, pass rush, sack classification, a deterministic passing arc, swept catch checks, per-play forward-pass eligibility, explicit ball states, a basic offensive drive, downs, yards-to-go, touchdown scoring, sack, tackle, incomplete, and out-of-bounds outcomes, turnover-on-downs reset, exact dead-ball spotting with three-lane snap placement, signed yardage, moving line of scrimmage, first-down marker, final-score game over, delayed reset, a preserved tactical orthographic camera, a behind-the-offense perspective camera, an optional cinematic broadcast camera, optional cinematic orbit shots, normal-game low-density crowd visuals and reactions in the broadcast profile, and full-cinematics fan cutaways. `GamePresentationRuntime` owns the shared presentation-event stream and coordinates existing audio, commentary, crowd, hold, and camera systems without determining gameplay results. Exhibition starts include a runtime pregame broadcast presentation phase with data-defined stadium/matchup/team/weather shots, presentation-only warmup groups, the user-team starting quarterback warmup spotlight, lower thirds, pre-rendered pregame announcer clips when available, a serialized coin-toss setup/result sequence, title-music duck/fade, crowd ambience, skip handling, F1/debug readback, and no gameplay or clock advancement until playable pre-snap. Audio, announcer audio, crowd presentation, and cinematic presentation are planned product scope. The current audio implementation includes offline ElevenLabs production tooling, a generated 15-asset football SFX/crowd pack using `eleven_text_to_sound_v2`, a generated 27-line prototype gameplay announcer voice pack with exact captions, a generated 49-line pregame broadcast voice pack with welcome, matchup, weather, quarterback spotlight, coin toss, and kickoff clips, three generated `music_v2` Football JS title-theme candidates with a provisional stable selection, typed announcer and pregame script catalogs, a prototype fictional announcer identity, a Node-only voice-design preview workflow, `npm run audio:verify`, a browser runtime mixer, explicit user-gesture audio unlock, a streamed title-music bus for the selected `football-js-title` asset, a read-only presentation event bridge with stable event IDs, ramped crowd ambience crossfades, deterministic result reactions, serialized pre-rendered announcer commentary with priority/cooldown logic, runtime pregame speech sequencing, optional exact-script captions, crowd ducking during speech, page-activity suppression, and audio/commentary/pregame debug history and memory diagnostics. The current brand implementation includes a central `Football JS` brand config, offline GPT Image 2 generation tooling, four generated provisional WebP candidates, a generated brand gallery, a selected title background, a selected emblem, and a title-music ID pointing to the provisional stable music asset. The current crowd implementation includes normal-game InstancedMesh visual crowd presentation through the broadcast/custom settings path, deterministic result reactions, a presentation hardening audit matrix, `?crowdPreview=1`, stadium-seat-layout-derived placement, InstancedMesh low-poly spectators, near/far detail tiers, preview camera controls, and benchmark reporting for 500, 2,000, 5,000, and 10,000 spectators. The current sideline implementation includes presentation-only instanced low-poly sideline teams and static tunnel tableau subjects under broadcast/custom settings, with mathematical zones, deterministic team colors and skin tones, bounded draw calls, and F1/debug metrics. Spatial audio, gameplay or dynamic music, live text generation, kickoff implementation, receive/defer choice, additional pregame segments beyond warmups and the QB showcase, detailed stadium polish, and final brand/legal approval remain future work. The `?formationPreview=7v7` and `?formationPreview=11v11` modes are static development staging tools for formation validation, rendering, camera framing, and presentation checks; they must not run AI or begin a live play. `?crowdPreview=1` is also a development-only mode and must not run gameplay AI.

## Current Non-Goals And Future Scope

- Presentation future scope: detailed stadium architecture, detailed crowd choreography, coaches, chain crew, advertisements, weather, field degradation, turf redesign, animated scoreboard video, and broader stadium polish are deferred product work, not permanent exclusions. The current presentation work includes an active mathematical stadium bowl, optional normal-game visual crowd presentation, presentation-only sideline teams and tunnel staging, and the development-only rendering and memory preview.
- Roster future scope: active 11v11 gameplay currently has Inside Zone 11 and Spread Quick 11 plus fictional starter identities and specialists. Additional 11v11 plays, full special teams behavior, depth-chart editing, substitutions, injuries, recruiting, roster tiers beyond the current 11v11 prototype and maintained 7v7/5v5 regression modes, player switching, and broader formation families are deferred.
- Assets and animation future scope: imported full-body player models, imported head models, facial features, hair, skeletal animation, quarterback animation, scramble animation, tackling animation, celebration animation, and center or snap animation are deferred. The current prototype intentionally uses procedural low-poly silhouettes, procedural heads, a procedural football, and the reusable low-poly helmet.
- Play calling and setup: no large playbook menu, final logo package, real team logos, audibles, defensive play selection, route editor, procedural play generation, hot routes, depth-chart editing, season mode, save slots, or menus beyond the current title/setup flow, team customization panel, roster preview, pause settings panel, pre-snap play cards, and minimal HUD/debug displays.
- Passing and ball outcomes: no interceptions, fumbles, loose-ball physics, manual aiming, pass-type selection, pump fake, illegal-forward-pass penalty, officiating authority or penalty logic, user-controlled catch mechanic, contested-catch ratings, or quarterback ratings.
- Blocking and tackling: no offensive linemen rules, holding penalties, pancake blocks, double-team blocks, pulling guards, diving tackles, advanced pursuit/pathfinding library, or physics-driven contact.
- Game structure future scope: no playable defense, playable field goals, playable punts or kickoffs, overtime, timeouts, NFL clock-stoppage rules, play clock, penalties, season modes, or franchise systems. The current Exhibition shell already includes four quarters, user score, opponent score, halftime, abstract punts, simulated opponent possessions, and final score.
- Controls and camera: no sprinting, stamina, freely rotating camera, camera-relative controls, instant replay, replay recording, slow motion, camera collision, or camera redesign beyond the current tactical, offense-perspective, cinematic broadcast, optional orbit-shot, and optional fan-cutaway presentation modes.
- Audio future scope: no spatial player sounds, gameplay music, dynamic music, live text generation, conversational agents, second commentator, or browser-side ElevenLabs calls in runtime gameplay. The first runtime MP3 pack, pregame voice pack, and provisional title-theme candidates are generated and verified locally; the title screen now streams the selected local theme after user gesture unlock and the Exhibition pregame sequence can play local pregame speech assets. Future audio work should preserve existing approved assets unless replacement is explicit.
- Simulation architecture: no force-based physics, ragdoll physics, general-purpose physics engine, advanced AI rewrite, or unrelated refactoring.

Stop after the current milestone unless the user explicitly asks for the next feature.

## Versioning And Release Workflow

- For every user request, decide whether the change is `major`, `minor`, or `patch` before editing.
- Use `major` for incompatible gameplay, API, save-data, or architecture changes.
- Use `minor` for new game features, scenes, systems, or user-visible capabilities that preserve existing behavior.
- Use `patch` for fixes, small polish, documentation corrections, and test-only updates.
- Bump `package.json` for each completed request.
- Update `CHANGELOG.md` in the same change set with the date, version, and user-visible summary.
- Commit and push each independent request individually. Do not hold completed work to batch later.

## Implementation Rules

- Keep `src/main.ts` as a thin bootstrap. Application lifecycle ownership belongs under `src/app`, with `FootballApplication` coordinating construction/start/stop only.
- Use the existing app lifecycle owners for scene/rendering, bounded game loop timing, gameplay orchestration, stable-ID player visual reconciliation, presentation coordination, and development diagnostics rather than adding new top-level orchestration to `main.ts`.
- Keep field construction exposed through `src/field.ts`, with rendering/layout/resource details in focused `src/field/*` modules.
- Keep authoritative field dimensions, paint widths, playable bounds, field bounds, and plain field layout data in `src/fieldSpec.ts`.
- `src/fieldSpec.ts` must stay free of Three.js imports.
- Generate rendered field-marking dimensions and positions from the field spec rather than repeating raw field dimensions in renderer code.
- Keep painted field markings fully inside the field surface; boundary paint outer edges should align with field outer edges, and transverse internal markings should stop at the inner sideline paint edges.
- Batch static field markings by shared material where practical; keep line of scrimmage and first-down line separate because they are dynamic.
- Keep field material ownership, geometry construction, marking layout, dynamic markers, and disposal responsibilities separated in the existing `src/field/*` modules.
- Field presentation meshes such as turf bands, yard numbers, goalposts, sideline apron, team boxes, and surrounding ground must remain presentation-only and must not influence collision, spotting, scoring, or boundaries.
- Keep yard numbers sideline-oriented and goalposts on the end lines at the back of the end zones.
- Painted hash marks and ball-spotting snap lanes must use the same authoritative hash X value from `src/fieldSpec.ts`.
- Normal-game presentation defaults belong behind `src/config/GameExperienceSettings.ts`. Do not independently resolve default camera, cinematics, visual crowd, crowd reactions, crowd audio, gameplay SFX, announcer, captions, route art, player motion, or playbook behavior in `main.ts`.
- Brand title, short title, selected image URLs, selected title-music ID, and announcer display name belong in `src/config/GameBrand.ts`. Do not hard-code new player-facing title strings, runtime brand image URLs, or title-music IDs in UI modules.
- The broadcast experience preset is the default normal-game profile: 11v11 playbook, offense camera, brief cinematics, mathematical stadium enabled, presentation-only officials enabled, presentation-only sideline teams and tunnel tableau enabled, low-density crowd visuals, crowd reactions, runtime audio, crowd audio, announcer enabled, captions disabled, route art enabled, player motion enabled, and Adaptive 60 FPS quality mode.
- The performance preset keeps the offense camera, mathematical stadium, and player motion while disabling expensive visual presentation: cinematics, visual crowd, crowd reactions, sideline teams, tunnel tableau, and officials. Performance may simplify stadium presentation by disabling upper-tier geometry and image materials.
- Runtime quality modes are `adaptive60`, `lockedBroadcast`, and `lockedPerformance`. Adaptive quality may adjust render pixel ratio and reduce crowd presentation only through `src/performance/*`, `src/app/SceneRuntime.ts`, and `src/app/PresentationRuntime.ts`. It must never change player count, roster, gameplay simulation, collision precision, ball trajectory, route progression, blocking assignment, score, drive state, controlled-player input, or camera focus targets.
- Crowd allocation or rebuild changes caused by quality transitions must wait for title/setup, pre-snap, dead-ball, or game-over boundaries. Never rebuild the crowd during a live run, pass, pass flight, catch, or scramble.
- Query parameters are explicit development/session overrides and must not rewrite persisted experience settings.
- Keep `formationPreview`, `crowdPreview`, `routeAudit`, `passAudit`, `presentationAudit`, `shotPreview`, and `appearanceAudit` development-only and opt-in.
- The title/setup flow belongs under `src/ui`. It should compose resolved `GameExperienceSettings`, not create a competing settings source.
- A normal launch should show the title screen before gameplay. Direct query and development preview/audit paths may still enter their requested mode directly for tests and debugging.
- Start Game must count as the audio-unlock user gesture, apply the selected settings, and select or construct the chosen playbook. Exhibition starts may enter `pregamePresentation` first; gameplay HUD and play cards should appear only after the intro completes or is skipped into playable pre-snap.
- While the title screen or pause settings panel is active, the score clock, gameplay AI, play controls, presentation result events, and play cards must not advance behind the UI.
- While `pregamePresentation` is active, the match clock, score-attack clock, gameplay AI, snap input, presentation result events, HUD, and play cards must not advance. `Space` or `Enter` may skip the intro and must not immediately snap the ball.
- Escape may open the pause settings panel only during pre-snap or dead-ball states. Live play should remain responsive and uninterrupted.
- Settings changed from the pause panel may apply presentation and audio changes safely. Game mode changes require returning to the title screen.
- Team profiles, uniform palettes, persisted customizations, and team-theme application belong under `src/teams`. Do not scatter new authoritative team colors across HUD, field, player, helmet, crowd, or play-card modules.
- Team profiles must be fictional. Do not use real professional or college team names, logos, or exact uniform designs.
- Persisted team colors must normalize to six-digit hex strings. Invalid saved colors should migrate to starter profile defaults rather than crashing startup.
- The same resolved team theme should drive player jerseys, shoulder pads, pants, helmets, faceguards, HUD/scorebug colors, play-card markers, end-zone presentation, crowd accent distribution where supported, and future stadium accent fields.
- Team and uniform changes during an active match should require returning to the title/setup screen rather than partially recoloring live gameplay.
- Skin tone must remain independent from team, role, selected receiver, play, camera, reset, and random state. Role-specific colors remain available only through explicit debug paths such as `?debugRoleColors=1`.
- Player and helmet material caches should remain keyed by shared team/uniform components. Do not create unique materials for every player when applying team customization.
- Roster identity belongs under `src/roster`. Use fictional names only, keep jersey numbers unique within each team roster, validate position-number conventions as warnings, and keep roster IDs separate from gameplay player IDs.
- Active lineup binding must map stable gameplay player IDs to roster player IDs without replacing IDs used by gameplay, AI, collision, routes, presentation events, or tests.
- Setup roster preview belongs under `src/ui` and should consume the same active lineup binding used by presentation. Do not create a second manually entered lineup list.
- Controlled-player and selected-receiver labels belong under `src/presentation/ControlledPlayerLabel.ts`. They must be pooled, presentation-only, cached by roster player ID, and hidden during title/menu/game-over or distracting presentation shots.
- Do not create one always-visible label per player, redraw canvas textures every frame, or let labels affect gameplay collision, pass targeting, AI, camera focus, or possession.
- Roster-label debug output may report gameplay ID, roster ID, name, number, position, label position, and visibility reason, but it must never mutate gameplay snapshots.
- Missing optional generated audio should not produce player-facing startup errors; show detailed asset readiness only through development/debug surfaces.
- Stadium presentation belongs under `src/stadium`. Keep stadium dimensions in `StadiumSpec`, stadium path sampling in `StadiumPath`, row/seat layout in `StadiumLayout` and `SeatLayout`, mesh construction in `StadiumGeometryBuilder`, materials/textures in `StadiumMaterialLibrary`, and scene lifecycle in `StadiumController`.
- Stadium math modules must stay independent from gameplay state. Stadium and crowd presentation must not change collision, field bounds, ball spotting, scoring, possession, drive rules, or player AI.
- Crowd placement must consume `SeatLayout` transforms from the stadium. Do not restore a separate sideline/end-zone crowd placement formula.
- Do not create individual seat meshes. Batch seating surfaces by tier/material and keep spectators as `InstancedMesh` resources.
- Stadium geometry must remain outside the protected field/apron bounds. Invalid stadium geometry should be caught by tests rather than hidden with clamps.
- Stadium image textures are prototype presentation assets only. Load them once, use sRGB color space for color textures, limit anisotropy to renderer capability, and provide simple-material fallbacks.
- Football officials live under `src/officials` and are presentation entities only. They may align to the authoritative ball position, line of scrimmage, direction of play, sidelines, field bounds, and dead-ball spot, but they must never enter gameplay rosters, collision queries, AI, scoring, possession, catch rules, penalties, ball spotting, or drive logic.
- Official visual resources should use shared low-poly primitive geometry/materials and must be disposable as one presentation-owned controller. Officials may expose debug labels and snapshots, but those diagnostics must never mutate gameplay snapshots.
- Touchdown official signals are visual feedback only and may respond only after an authoritative touchdown result already exists in gameplay state.
- Sideline and tunnel team presentation lives under `src/presentation/teams`. These subjects may read field bounds, team-box zones, stadium tunnel anchors, team themes, and lineup appearance IDs, but they must never enter gameplay rosters, collision queries, AI, blocking, coverage, possession, scoring, or assignment logic.
- Sideline and tunnel visuals should use instanced/shared low-poly primitive resources with bounded draw calls, deterministic skin tones, precomputed pose matrices, and explicit lifecycle disposal. Do not create independent animated Object3D hierarchies for anonymous sideline reserves.
- Sideline debug snapshots may report active counts, zones, draw calls, triangles, instance-buffer bytes, update frequency, and visible mesh counts, but must never mutate gameplay snapshots.
- Use primitive Three.js geometry and simple materials during graybox work.
- Keep input, simulation, and visual synchronization in separate modules.
- All gameplay players use the common player model with stable ID, team, role, position, velocity, facing, collision radius, and current state.
- Default active formations use `ELEVEN_ON_ELEVEN_ROSTER` and `src/elevenOnElevenFormation.ts`. They preserve the shared 7v7 IDs where practical, add tackles, tight end, slot, linebackers, and strong safety, and track football-position metadata separately from behavior-oriented `PlayerRole`.
- The stable 14-player 7v7 roster remains available only through explicit development/regression selection: `?playbook=7v7` or `7v7 Development Regression Mode`.
- `?formationPreview=11v11` must remain a development-only static preview. It may resolve, validate, render, label, measure, and frame twenty-two players, but must not run AI or snap into live play.
- `?playbook=11v11` is the normal default playbook and currently contains Inside Zone 11 and Spread Quick 11. Future playbook additions should target 11v11 first unless they are explicit regression fixes.
- Inside Zone 11 must use `ELEVEN_ON_ELEVEN_ROSTER`, the validated 11v11 formation, running-back possession, a quarterback decoy, explicit one-to-one run-blocking assignments, unblocked safeties, and the shared result/presentation systems without roster-specific event duplication.
- Spread Quick 11 must use the validated 11v11 formation, quarterback possession, five ordered eligible receivers, explicit pass-protection assignments, explicit coverage assignments, deep safety help from receiving-threat bounds, route-aware targeting, swept catch checks, sack classification, incompletion handling, and shared result/presentation systems without roster-specific event duplication.
- 11v11 formation preview controls are `1` left hash, `2` middle, `3` right hash, and `4` mirror preferred formation side.
- The legacy 5v5 roster remains available only through explicit development/regression selection: `?playbook=5v5` or `5v5 Legacy Regression Mode`.
- Initial formations belong in semantic data resolved through `src/formationLayout.ts`, not hard-coded mesh positions or independent per-play clamps.
- Formation preview positions must resolve through the semantic formation system for each snap lane and must fail validation rather than clamping invalid geometry.
- Formation data should separate formation position, pre-snap facing, post-snap movement direction, blocking targets, route targets, and coverage assignments.
- Play definitions belong in data and must stay independent from Three.js scene objects.

- SVG play-card coordinate mapping belongs in pure transformation code and should not mutate gameplay state.
- Pointer/tap play-card selection must feed into the same request path as keyboard play selection; DOM event handlers must not directly mutate the gameplay model.
- Play selection is allowed during pre-snap only; resetting preserves the selected play.
- The gameplay model owns player position, velocity, facing, and blocking engagement state; Three.js meshes only display that state.
- The gameplay model owns play state, ball possession, pass state, and pass flight data; the ball mesh is never authoritative.
- The gameplay model owns play results, start spots, exact dead-ball spots, normalized next snap spots, snap lanes, yards gained, and scoring team data; UI and meshes only display that state.

- Goal-line detection and scoring must use gameplay coordinates, not mesh positions.
- Sideline and dead-ball spotting must use gameplay coordinates, not mesh positions.

- Receiver simulation, graphical play cards, and on-field route art must consume the same `ResolvedReceiverRoute` objects or route-resolution helpers. Do not maintain separate route-art or play-card coordinates.
